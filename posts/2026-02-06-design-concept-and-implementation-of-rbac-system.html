rbac系统的设计理念与实现

📅 2026/2/7 ✍️ 字数：3436
rbacnestjs后端开发

前言

有了AI后，代码一键生成，很方便。基本的框架就有了。针对业务需求，AI也能即时输出，给出方案，给出代码，实现不在话下。一些手动写代码的时代，是要花三天三夜找出来的bug，也能被AI快速定位。但是，AI用多了，针对不同的业务需求，随着时间推移，系统会变的难以维护。都是AI生成的代码的项目，同样会变乱，这里有时间的因素，上下文的因素。其实即使是人，也会随着时间推移，代码越来越臃肿。但是有个变量更容易使人焦虑，和以前不同，人可能忽视或者完全忘记AI这些代码的细节，因为自己写出来才是刻骨铭心，捡起来也容易。那如何不让一次次代码不变乱？还得靠人，人还是要懂代码，而且要以上帝视角去懂代码，懂代码的设计，除了懂需求，还要懂需求和代码语言逻辑，特别是顶层的需求和顶层视角下逻辑关系，一次次审视代码，尽量不让代码生成到天边去。

怎么实现一个rbac系统？

问：rbac 全称是role base access control，基于角色的权限控制。它很普遍。现在想到实现rbac你会想到什么？

想： 首先想到的是角色是怎么授权的，我想要具象化些，首先从产品的角度来看，这个rbac需要做什么事情啦。我们给这个它上业务，是用来做电商门店业务的。需要做一个门后台系统。它有用户管理，订单管理，商品管理。这是最简单的。基于角色，老板和普通员工，店长，普通员工看不到用户管理，订单管理，授权管理。非老板看不到授权管理，老板可以看到全部。我们系统rbac能解决不同用户看到不同目录。这就是角色最直观的控制了。

问：首先要实现，第一步了解需求，除了技术选型，就要定义数据设计，步骤就是如此，怎么从 业务场景清晰化 联想到数据库表结构？

想：从具体的场景到数据哭表结构有个通用的模式：确定有哪些实体。想想看，有用户，有角色，还有权限。确定自己属性。第二步就是他们之间有什么关系，其实都是多对多关系。这就意味着用户和角色，角色和权限之间需要关系表。这就推倒除了数据库表。

问： 除了权限控制，还有登录注册吧？

想：其实和rbac没有关系了，是个独立的东西。

问：登录的时候都有身份验证，都会用到token，为什么需要token?

想： token 有点像号码牌，每个人都有自己的号码牌，后台可以识别。进化到token，是有一定的历史的。

一开始的凭证是账号密码。但是不可能每次都是账号密码，疯狂暴露。

能不能第一次用账号密码，成功后有session_id 放在cookie里，以后每一次都用这个session_id证明用户登录过，后台已经session_id和用户信息一一对应

但是这种很难用于分布式，因为分布式系统需要同步用户的登录信息，比较麻烦。而且其他非浏览器没有session

在这个背景下有了token，如果这种token只是一个随机串，本质上它还是session_id。

这是jwt出现了，叫做无状态的token，它里面本身保护用户信息，不用再去查数据库。

后来又有access token+ refresh token，access token 有效期短，refresh token 反正http only cookie里。

问：access token+ refresh token 不是可以通过频繁刷新等同于获取到了永久token有效吗？

想：Refresh Token 有效期是7天，超过了就要重新登录。假如Refresh Token 每次刷新不一样。那么自用户登录器，可能30天后必须要重新登录。总之是有绝对有效期的。不会永久。
