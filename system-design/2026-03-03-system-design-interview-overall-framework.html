系统设计面试--[总体框架]

📅 2026/3/5 ✍️ 字数：2955
系统设计面试
题外话

发现长久以来，自从准备考研之后，学习就变的不是那么哇塞（开心）了。算起来就差不多有个10几年了，好像不会学了一样，即使在一个绝对安静的环境，时长发生看着看着学习资料，去看别的东西去了。到今天为止，我觉得就是，学习目标激不起持续性的专注，激不起的原因就是学着不知道为了什么。学语言，学技术，干嘛呢？想来想去，后面有了个答案，就是为了钱。学语言，就是个敲门砖，可以接触国外的工作。像今天的系统设计是为了，我对技术面试有更多的把握，心里低。这些工作都是为了一个字，钱！，拜托现在底层奴工，工资只能糊口的状态。好像只有钱，可以增加学习的自我效果感。

有时候啊，在想，人生都在学习，多没劲啊，每天都在写代码，每天都在奋斗的路上，这样的人生都是我要的吗？心里的回答肯定不是！那我要怎么做才开心。想着就是不报任何期待，努力做就行，去日本就是试试学号语言找找看能找到什么工作。学英语，学技术，看看自己能走到哪里，看看能不能找到远程的，好价钱的工作，仅此而已。

总而言之，学习的路途上，应该是带着自己的爱，带着自己的喜欢才是长久之路。

前言

很多程序员的都会掉了一种模糊陷阱，特别是前端。总觉得自己缺什么，有时候甚至知道自己缺什么，不知道具体怎么补。我觉得这里面有低行动力的原因，还有对缺的东西带有恐惧而却步的原因也有。

我以前对系统设计这一块真的迷茫。没有系统的学过。当了这么多年的程序员，好像还是门外汉的感觉。这种感觉很可怕，久而久之，成为了我的无助来源，无限自我否定。所以打算给自己开个专栏学写系统设计50篇，加上自己做的项目列在线上，包装下自己，去面试看看效果如何，到时候，再缺什么补什么。

AI时代下的系统设计

现在AI发达的可以提供整个系统的设计方案了，所以学还有什么作用呢？有了openClaw，甚至整个平台都可以被AI接管，无需人操作。但是所实话，现在是2026年3月份，截止这个时间点，AI还是没有高效到没有任何缺点，依然存在不符合实际情况的问题，即使是专门领域的AI也有这种问题，这就引申出一个问题，需要人去审核，让AI做成符合实际场景的事情。这里就需要专业的人能够从顶端审视我们的代码，系统，现状。

目前大部分公司其实使用全自动的AI去应用到自己实际业务中，依然存在成本问题，除了上面说的和实际契合度的问题。小企业可能根本不需要太多这种AI完全自动化，如何出现问题的时候管理AI能力，它们可能不想花太多力量在这里。

当然使用AI递归式学习系统设计，是很有意思，你会发现，可以快速扩展自己的知识面，AI是个不知疲倦的老师，无所不包，无所不答。非常时候一个人无边界，无顾忌的去学习。然而自己去总结，再输出。

系统面试的总体框架是什么？

1、系统面试没有标准答案

这点有点缓解焦虑的效果了，因为很多时候一听到，设计一个某某系统的时候，就感觉特别大，或者听都没听过，不知道怎么回答，马上心跳加速，脸色尴尬，整个面试就完全不在状态。其实总觉得要一个标准答案，就是在自虐。要做到尽善尽美完全不可能，因为一讲一大堆，面试的时间很有限。所有放轻松，结合正常范式和当下节奏，自己的知识面有答完就可以了。

2、你的系统面试提问框架。

当面试抛出设计某某系统的时候，一开始要做的。就是要搞清楚范围。这个范围总结起来有这么几项。

  1. 这个系统是做什么，或者说核心功能有哪些，也可以自己列出核心功能，让面试官确认一下。

    面试的时候，其实只要回答的就是核心功能的设计了。在这个范围内取考虑。

  2. 这个系统用户规模？比如月活这些比较场景，总的用户量等等

    不同的用户规模要考虑对系统设计不同的扩展，这些扩展也是考虑到设计之中的

  3. 这个系统的数据是什么样的？

    系统的不同数据就要考虑用什么数据库，用什么服务器，用什么存储媒介回好点了。

  4. 这个系统的其他非功能需求和限制

    有些系统除了功能需求还有别的需求，比如数据一致性、数据实时性等等，预算有限制，哪怕是技术栈可能有限制。那这也要考虑到设计之中。

以上呢就是具体的问清楚框架。

3、画高层级图的框架

高层级别的组件总结起来一般就是这么几个，不同系统肯定有加有减。

  1. Actors/Clients：人/客户端/外部系统
  2. Edge/Entry：DNS（可选）、CDN（按需）、LB/API Gateway（常见必画）
  3. Core Services（盒子）：按职责拆分的业务服务
  4. Data Stores：DB / Index / Object Storage（至少标清主数据落点）
  5. Cache（按需）：降低延迟、扛读

这几个是必须的。涉及到其他常见的比如消息队列啥的等，需要特意提出来的。

现在看到这里你可能还是有些懵，到底不同的高层级别图，具体要怎么画，这里是比较场景的几个组件，但是具体画感觉还是吃力。先不着急，因为内容还有很多，怎么画，其实还涉及到经验积累，也许很难一蹴而就。如果看了几章还是不懂，再回过头来。以免立刻就陷入很复杂，太揪心的局面。

4、总体框架

前面提到，第一步是提问题，先不着急回答。提问题，按照核心功能是什么，数据量，用户量，数据类型，非功能需求和限制 这几个面向去问。有些系统有自己特殊的因素，也需要提问题去问。

第二步，就是把高层级草图画出来。这一块首先是自己画，上面也提到到了基本的几个组件，入口，负载均衡，核心服务，数据存储 这几个都是要的。

第三步，就是和面试官讨论，这个图怎么样，从面试官那里获得了对高层设计的反馈。面试官可能会在这个图上再深入提问题，不能排查会问很细节的问题，不对往深里问。但是理想的面试官会问，如何扩展这个图，也就是如何扩展这个系统，或者遇到问题的时候，如何改变系统的某一部分适合实际的需求等。

写在最后

作者在最后总结了一份该做和不该做清单

这一篇我们划定了一个系统面试的基本框架，这个框架到了两层。第一层是总体框架。第二层。包括提问框架，画高级设计图的框架，讨论框架。后面都是在这个基础上去丰富。

系统设计还是得多看，即使没有实际操作。看的多了，在真的遇到了，会发现，你才能更胸有成竹的感觉。真的是相见恨晚啊
